# OpenConfig Network Automation Lab

A home-lab pipeline that uses NetBox as the Source of Truth (SoT), translates
device data into validated OpenConfig payloads via Pyangbind, and deploys
those configs to virtual network devices over NETCONF using Ansible.

## Architecture

```
[ NetBox (Docker) ]  --REST/GraphQL-->  [ Translator App (Docker) ]
   Inventory & IPAM                       - netbox_client.py (pynetbox)
   "Source of Truth"                      - transformer.py (NetBox -> OpenConfig)
                                           - Pyangbind validation (oc_bindings.py)
                                           - main.py: webhook receiver / CLI
                                                  |
                                                  v  (writes validated JSON/XML
                                                      payloads + Ansible vars)
                                           [ Ansible (NETCONF) ]
                                                  |
                                                  v
                                  [ Virtual Devices (containerlab + Nokia SR Linux) ]
```

### Why this split?

- **NetBox** owns human intent: device roles, IPAM, cabling, VLANs. It is
  *the* Source of Truth (backed by Postgres).
- **Pyangbind** is *not* a SoT. It's a stateless, in-memory schema validator
  generated directly from the OpenConfig YANG models. It guarantees that
  whatever NetBox gives you is structurally and semantically valid before it
  ever touches a device.
- **The translator app** is the glue: a small Python service with one job —
  map pynetbox objects into pyangbind OpenConfig objects (`transformer.py`),
  validate, and serialize to JSON/XML.
- **Ansible + NETCONF** takes the validated payload and pushes it to the
  device's `<edit-config>` datastore. Ansible gives you idempotency, dry-run
  diffs, inventory management, and a familiar operational model.
- **containerlab + Nokia SR Linux** gives you free, NETCONF/OpenConfig-native
  virtual routers to test against, with no vendor licensing required.

## Project Layout

```
openconfig-automation/
├── docker-compose.yml          # NetBox + translator app
├── translator/
│   ├── Dockerfile
│   ├── requirements.txt
│   ├── build_bindings.sh       # one-time: clone OpenConfig YANG, compile pyangbind classes
│   ├── netbox_client.py        # thin pynetbox wrapper
│   ├── transformer.py          # NetBox object -> Pyangbind OpenConfig object
│   ├── main.py                 # FastAPI webhook receiver + CLI entrypoint
│   └── oc_bindings/             # generated by build_bindings.sh (gitignored)
├── ansible/
│   ├── ansible.cfg
│   ├── inventory.yml
│   ├── group_vars/all.yml
│   └── playbooks/deploy_interfaces.yml
├── containerlab/
│   └── topology.clab.yml       # virtual SR Linux devices
├── .env.example
└── .gitignore
```

## Setup Order

### 1. Bring up NetBox

This compose file expects the official `netbox-docker` project's compose file
to be merged in (or run as a separate stack on the same Docker network — see
`docker-compose.yml` comments for both options).

```bash
git clone -b release https://github.com/netbox-community/netbox-docker.git
```

### 2. Compile the OpenConfig -> Pyangbind bindings (one-time)

Run this once on your machine, or as part of the translator image build:

```bash
cd translator
./build_bindings.sh
```

This clones `openconfig/public` from GitHub and compiles the interfaces,
VLAN, and routing-policy YANG models into `oc_bindings/`. The output is large
and machine-generated, so it's gitignored — re-run the script whenever you
need it (this is the same "Phase 1: Build Time" step described in the design
discussion).

### 3. Stand up the virtual network with containerlab

containerlab needs to talk to the host Docker daemon directly (it's not
typically nested inside another container). Install it on your host:

```bash
bash -c "$(curl -sL https://get.containerlab.dev)"
cd containerlab
sudo containerlab deploy -t topology.clab.yml
```

This brings up two Nokia SR Linux nodes with NETCONF (port 830) exposed.

### 4. Bring up NetBox + the translator app

```bash
docker compose up -d
```

### 5. Populate NetBox

Create your devices, interfaces, and IPs in NetBox (manually via the UI for
now, or via `pynetbox` scripts later).

### 6. Run the translator

```bash
docker compose exec translator python main.py --device spine-01
```

This pulls the device from NetBox, runs it through `transformer.py`, validates
it against `oc_bindings`, and writes:

- `output/spine-01.json` — the validated OpenConfig payload
- `ansible/host_vars/spine-01.yml` — Ansible variables derived from the payload

### 7. Deploy with Ansible over NETCONF

```bash
cd ansible
ansible-playbook playbooks/deploy_interfaces.yml -l spine-01
```

## What goes into Git vs not

| In Git | Not in Git |
|---|---|
| `docker-compose.yml`, Dockerfiles | Running containers / images |
| `transformer.py`, `main.py`, `netbox_client.py` | NetBox Postgres data volume |
| `requirements.txt` | `.env` (secrets, API tokens) |
| `build_bindings.sh` | `oc_bindings/` (generated) |
| Ansible playbooks/inventory (no secrets) | Ansible vault password files |
| `containerlab/topology.clab.yml` | containerlab lab state / certs |

## Roadmap / Suggested Order of Automation

1. Interfaces + IP addressing (this scaffold's example)
2. VLANs (the `convert_netbox_vlan_to_openconfig` example from the design doc)
3. BGP (ASNs, neighbors, policies)

Each of these gets its own function in `transformer.py` and its own
compiled YANG module in `build_bindings.sh`.
